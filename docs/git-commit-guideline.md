# Git Commit 撰寫規範

## 目的

為了讓專案後續的提交紀錄更一致、可讀且便於回顧，本文件整理一套統一的 git commit 撰寫格式。

這份規範特別適合：

- 功能開發提交
- 架構調整提交
- 文件補充提交
- bug fix 提交
- 初次建立基線或里程碑提交

## 核心原則

- 主旨要短，直接說明這次提交完成了什麼
- 內文要能回答「為什麼做」與「做了哪些事」
- 條列內容要可快速掃讀
- 若有驗證結果，應明確寫出
- 若提交較大，建議補充此提交的定位與後續用途

## 建議格式

```text
<簡潔主旨>

本次提交說明一段，交代背景與目的。

已完成項目如下：

- 項目一
- 項目二
- 項目三

驗證結果：
- 測試或建置結果

後續說明：
- 若有後續方向再補一段
```

## 各段落用途

### 1. 主旨

主旨建議：

- 使用一句話說清楚本次提交主題
- 優先描述結果，不要描述操作
- 盡量避免過長，但也不要短到失去資訊量

範例：

- `完成初版 macOS Ollama Chat 應用程式骨架與 Phase 1-4 實作`
- `補上聊天紀錄持久化與多會話切換功能`
- `修正聊天輸入框焦點異常與視窗啟用問題`

### 2. 背景與目的說明

這一段建議用 1 到 3 行說明：

- 這次提交的背景
- 為何需要這次變更
- 本次提交在整體開發流程中的定位

範例：

```text
本次提交主要補上本地聊天應用程式的關鍵互動能力，
讓專案從靜態骨架提升為可實際測試 Ollama 對話流程的版本。
```

### 3. 已完成項目

這一段是最重要的掃讀區塊。

建議：

- 使用條列
- 每一點對應一個具體變更
- 一個項目只說一件事
- 優先描述使用者可感知的成果與工程上有價值的結構變更

範例：

```text
已完成項目如下：

- 新增 Ollama API client，支援模型清單與聊天串流。
- 建立聊天畫面、側邊欄與設定視窗。
- 加入本地 JSON 儲存機制，保存聊天紀錄與設定。
```

### 4. 驗證結果

若這次提交有執行驗證，建議保留此段。

適合寫入的內容：

- `已執行 swift build，確認可成功編譯`
- `已執行單元測試，全部通過`
- `已手動驗證聊天送出、串流回覆與停止生成流程`

若尚未驗證，也可誠實寫明：

- `尚未進行完整 UI 手動驗證`
- `尚未補上自動化測試`

### 5. 後續說明

當提交較大，或只是某個階段性基線時，建議補這一段。

可說明：

- 這次提交是基線、過渡版本或里程碑
- 後續預計補什麼
- 哪些部分刻意延後

## 推薦模板

### 一般功能提交模板

```text
<功能主旨>

本次提交主要補上 <背景或目的>，
讓 <功能/模組> 能夠支援 <具體結果>。

已完成項目如下：

- ...
- ...
- ...

驗證結果：
- ...

後續說明：
- ...
```

### Bug Fix 提交模板

```text
修正 <問題主旨>

本次提交針對 <異常現象> 進行修正，
避免 <使用者影響或系統影響>。

已完成項目如下：

- 調整 ...
- 修正 ...
- 補上 ...

驗證結果：
- 已重現原問題並確認修正後行為正常
- 已執行 ...
```

### 文件提交模板

```text
補充 <文件主題> 說明

本次提交新增或更新 <文件名稱>，
整理 <範圍> 的使用方式、規則或設計方向。

已完成項目如下：

- 補充 ...
- 整理 ...
- 明確化 ...
```

## 建議避免的寫法

以下寫法可讀性通常較差：

- 只寫 `update`
- 只寫 `fix bug`
- 主旨過長，沒有分段
- 內文只有流水帳，沒有條列
- 使用字面上的 `\n` 當換行
- 提交說明只描述「執行了什麼指令」，沒有說明結果

## 實務建議

### 1. 不要把 `\n` 寫進 commit message

若要寫多行 commit message，建議使用以下方式：

- 多個 `-m`
- `git commit -F <file>`
- heredoc

例如：

```bash
git commit -F - <<'EOF'
修正聊天輸入框焦點異常

本次提交修正 app 視窗啟動後未正確取得前景焦點的問題，
避免使用者輸入文字時，鍵盤事件仍落到前一個應用程式。

已完成項目如下：

- 加入 app 啟動時的前景啟用處理。
- 補上主要聊天輸入框的焦點設定。
- 更新文件說明此問題與驗證方式。

驗證結果：
- 已確認啟動 app 後可直接在聊天輸入框輸入文字。
EOF
```

### 2. 主旨與內文要分開看

主旨是給 `git log --oneline` 和歷史掃讀使用。  
內文是給未來回頭看細節時使用。

所以：

- 主旨要精簡
- 內文要有結構

### 3. 大提交更需要可讀性

當一次提交跨越多個模組、階段或功能時，應特別重視：

- 條列分段
- 驗證結果
- 此次提交的定位

## 建議採用慣例

本專案後續可統一採用以下順序：

1. 主旨
2. 背景與目的
3. 已完成項目
4. 驗證結果
5. 後續說明

## 總結

一份好的 commit message，應該讓未來的自己或協作者在打開 `git log` 時，可以快速理解：

- 這次改了什麼
- 為什麼改
- 改動範圍在哪
- 是否有驗證
- 這筆提交在整個專案中的位置

若後續沒有特殊情況，建議本專案提交訊息統一依本文件格式撰寫。
